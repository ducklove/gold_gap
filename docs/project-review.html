2.프로젝트 개요 · 아키텍처

국내 거래가와 환산 국제가의 괴리율(김치프리미엄)을 추적하는 대시보드입니다. 자산은 3종 — 금(KRX 금현물 411060 ETF ↔ COMEX 선물 GC=F / 런던 현물 XAU), 비트코인(업비트 ↔ BTC-USD), USDT(빗썸, 실패 시 업비트 ↔ USDT-USD). 데이터 보유량은 금 1,041포인트(2021-12~), BTC 1,833포인트(2021-06~), USDT 917포인트(2023-12~), data.json 약 340KB입니다.

핵심 데이터 흐름

  1. 정기 갱신(주 경로): cron(30분) → generate_data.py가 기존 data.json의 마지막 날짜−7일부터 증분 fetch → 병합 → 커밋·푸시 → Pages 재배포. GITHUB_TOKEN 푸시는 워크플로우를 재귀 트리거하지 않으므로 gh workflow run deploy.yml로 명시 트리거 — 플랫폼 제약을 정확히 이해한 구현입니다.
  2. 실패 시: 자산 단위 실패 → 해당 자산만 기존 데이터 유지 / 전체 실패 → 기존 data.json 폴백 (사이트가 죽지 않음).
  3. 브라우저 새로고침 버튼: api/data?force=1 시도(정적 호스팅에선 404) → 실패 시 Upbit·Binance·Bithumb·gold-api·환율 API를 브라우저에서 직접 호출해 오늘 포인트를 합성. 거래 시간대가 다른 자산의 현재가를 근사하는 실용적 설계지만, 마지막 종가와 실시간 시세를 섞어 만든 합성 포인트라는 사실이 어디에도 문서화되어 있지 않습니다.
참고: config.json은 현재 아무도 읽지 않습니다 config.json(자산 라벨·임계치·portfolioCodes)은 deploy.yml이 배포 산출물로 복사만 할 뿐, 이 리포의 어떤 코드도 참조하지 않습니다. 외부 포트폴리오 프로젝트와의 연동 계약으로 보이는데, 그렇다면 그 계약(스키마·소비자)을 README에 명시해야 합니다. 반대로 프론트엔드는 동일한 정보를 index.html 안의 ASSETS 객체에 별도로 하드코딩하고 있어 §4.3의 중복 문제로 이어집니다.

4.품질 평가

4.1 강점 — 유지해야 할 설계

  • 4단계 가용성: 증분 갱신 → 자산별 실패 격리 → 전체 실패 시 기존 데이터 폴백 → 브라우저 직접 시세. 무료 인프라에서 얻을 수 있는 최대 수준의 회복탄력성입니다.
  • 증분 업데이트 설계(7일 오버랩 재수집)로 외부 API 부하와 실행 시간을 최소화. 마지막 날짜 기준 시작점 산출, 겹치는 날짜는 새 데이터 우선 — 합리적인 규칙.
  • 도메인 정확성: 411060 ETF의 CU 구성(좌당 금 그램수)을 네이버 API에서 동적으로 조회해 원/g로 환산, 트로이온스 변환 상수 분리, 금 국제가를 선물/현물 듀얼 모드로 제공.
  • 일관된 수집 파이프라인: fetch → 컬럼 정규화 → 최신 quote upsert → 환산의 패턴이 모든 소스에 동일하게 적용되어 새 소스 추가의 멘탈 모델이 명확.
  • 프론트 완성도: URL 쿼리로 자산/금모드/테마/임베드 상태 공유 가능, aria 속성과 :focus-visible, 시스템 폰트 스택, 모바일 대응까지 — 1인 프로젝트 기준 상위권.
  • Python 코드 가독성: 함수가 짧고 이름이 정직하며, 한국어 docstring과 로깅이 일관됨.

4.3 구조적 부채 — 기능 추가를 막는 것들

① 도메인 로직 3중 구현 (가장 시급)

괴리율 공식과 고괴리 구간 탐지가 Python DataFrame 버전, Python list 버전, JavaScript 버전으로 각각 존재합니다. BUG-02가 보여주듯 이미 산식이 어긋났고, 임계치를 바꾸거나 "연속 N일 이상만 구간으로 인정" 같은 규칙을 추가하려면 3곳을 동시에 고쳐야 합니다.

② 자산 메타데이터 3곳 분산

임계치(금 5% / BTC 5% / USDT 3%)와 자산 라벨이 data_fetcher.THRESHOLDS, config.json, index.html ASSETS에 각각 존재합니다. 지금은 우연히 일치하지만 강제하는 장치가 없습니다.

③ 자산 추가 비용이 비선형

새 자산(예: ETH) 하나를 추가하려면 현재 4개 파일 7군데+를 수정해야 합니다: data_fetcher(fetcher + 오케스트레이터 + THRESHOLDS), generate_data(fetchers 리스트), index.html(ASSETS + 탭 버튼 + 금 전용 분기 확인), config.json. 금의 듀얼 모드(intl_modes)가 금 전용 특례로 구현되어 프론트 곳곳에 currentAsset !== 'gold' 분기를 남긴 것이 대표적 증상입니다.

④ 테스트·문서 부재

테스트 0개, README 0줄, LICENSE 없음. 특히 merge_asset_data(증분 병합)는 dict 누적 + 필수 필드 필터 + 모드 재귀 병합이 얽힌 가장 복잡한 함수인데 회귀 안전망이 전혀 없습니다. 50개 커밋 중 47개가 봇의 데이터 커밋이라 사람의 변경 이력으로 코드 의도를 추적하기도 어렵습니다.

  • 간단 백테스트: "괴리율 +X% 매도 / −Y% 매수" 규칙 시뮬레이션.
